Learning cycle - intelligence layer

Executive summary: 4/4 packages merged after 2 human gates. Pattern matches the v3 Knowledge Vault entry on regulator-first refactors. Confidence on next-cycle savings: HIGH.

Observations

  • O-1 - Engineers converged on EntityContext lookup pattern within first iteration.
    Evidence: All 4 packages use ctx.config(entity_id).<field>; zero rework loops.
  • O-2 - COBOL package consistently takes ~2x other languages.
    Evidence: WP-COBOL: 74+/22- lines but 3 QA tests vs 7-10 elsewhere (vault v5 pattern).

Patterns detected

  • Regulator-first refactor (vault v3) reduced churn by ~30%.
  • Pinning canonical entity values upfront (vault v2) prevented all QA test drift.

Recommendations (ranked)

  • P0 Auto-generate entity-*.yaml stubs from regulator code list before dispatch (Engineering Manager, ~4h)
    Cuts setup time by ~1h per engineer; lifts vault v6 from advisory to enforced.
  • P1 Add QA floor of 8 tests for COBOL packages (QA, ~1h)
    Vault v5 + v7 both flag this; we have repeated evidence.
  • P2 Move sample_diff fields out of LLM schema; render from real git diff (Platform, ~3h)
    Frees ~20%% of engineer token budget.

Guardrails for next cycle

  • Reject delegation_plan if any package lacks entity_config_format AND env_var_name
    Enforced by: t_eng_mgr_delegate schema validation
  • QA test count floor = max(8, residual_hardcodes_flagged * 2)
    Enforced by: t_qa_per_package post-check

Risk flags

  • medium Orphaned holding rows default to SG in V2 migration
    Mitigation: Surface as data-quality error in pre-flight script; ticket DASH-DEMO-FOLLOWUP.
  • low client.residency_region NOT NULL drops legacy DEFAULTs
    Mitigation: App team to backfill via entity_config on next deploy.
